# 채널 플레이북 — 상세페이지 / 랜딩페이지 (에셋+텍스트 레이어 모듈 기반)

> `문서 아카이브/SEO GEO/KDT_상세페이지_생성모듈_SEO정책_v3.md`를 특정 모듈(BDL/Tailwind)에 종속되지 않는 범용 규칙으로 일반화. **이미지 디자인 에셋 위에 텍스트를 얹는 구조**(카드뉴스형 랜딩, 노코드 빌더, 자체 CMS 등)를 쓰는 모든 상세페이지·랜딩페이지에 적용.

---

## 1. 전제 — 왜 이 유형이 특별 취급이 필요한가

일반 블로그는 에디터가 알아서 헤딩 태그를 파싱해주지만, **디자인 에셋을 배경/이미지로 깔고 그 위에 텍스트를 얹는 구조**는 개발/디자인 단계에서 태그를 잘못 쓰면 화면은 똑같이 보여도 검색엔진에는 텅 빈 페이지로 보일 위험이 크다. 자세한 원칙은 `../01-technical-seo/html-semantic-structure.md` 참조. 이 문서는 상세페이지 제작 워크플로우에 맞춘 **역할별 체크리스트**다.

---

## 2. 역할별 체크리스트

### 2-1. 기획(콘텐츠 작성자)이 준비할 것

| # | 준비물 | 설명 |
|---|---|---|
| 1 | 과정/상품별 키워드 목록 | 메인 1개 + 연관 5~8개 |
| 2 | 페이지 제목/설명 초안 | 55~60자 / 120~150자 |
| 3 | 섹션별 텍스트 원고 | 에셋 위에 얹을 모든 텍스트 — 검색엔진이 읽는 콘텐츠의 전부 |
| 4 | 에셋별 설명글(alt) | 모든 콘텐츠 이미지에 대한 20~50자 설명 |
| 5 | 태그 지정 가이드 | 어떤 텍스트가 H1/H2/H3/본문/링크인지 명시 (아래 §3 양식 사용) |
| 6 | CTA 문구 + 연결 주소 | 텍스트 링크 버튼으로(이미지 배너 아님) |
| 7 | 내부 링크 연결 지도 | 관련 상품/과정 페이지 연결 관계 |
| 8 | 구조화 데이터용 정보 | 과정명/기간/시작일/가격 등 — `../01-technical-seo/structured-data-jsonld.md`의 `Course`/`Product` 스키마용 |
| 9 | 종료/모집마감 처리 기준 | 숨김 vs 리다이렉트 |

**주의**: 에셋이 시각적 대부분을 차지하므로 텍스트 양이 일반 웹페이지보다 적을 수 있다 → 문장당 키워드 비중이 커진다. H1에 메인 키워드 필수, H2에 연관 키워드.

### 2-2. 디자인/에셋 제작자가 챙길 것

| # | 기준 | 규칙 |
|---|---|---|
| 1 | 이미지 파일명 | 영문 키워드 하이픈 연결 |
| 2 | 이미지 형식 | WebP 필수 |
| 3 | 이미지 용량 | 에셋당 300KB 이하 |
| 4 | 설명글 전달 | 이미지 납품 시 alt 텍스트 함께 전달 |
| 5 | 콘텐츠/장식 구분 | 납품 시 "콘텐츠용/장식용" 분류 표시 |

### 2-3. 개발자가 챙길 것

| # | 항목 | 설명 |
|---|---|---|
| 1 | 태그 선택 가능 여부 | 텍스트 입력 시 대제목/중제목/본문/링크 선택 → 자동으로 올바른 태그 생성 |
| 2 | img vs CSS 배경 구분 | 콘텐츠 이미지는 `<img>`, 장식만 CSS 배경 |
| 3 | alt 입력칸 | 비어있으면 경고 |
| 4 | CTA를 `<a>` 태그로 | onclick 전용 요소 금지 |
| 5 | `<section>` wrapper | `<div>` 대신 시맨틱 태그 |
| 6 | 제목 순서 자동 검증 | H1 2개 이상/H1 누락 등 자동 경고 |
| 7 | head 정보 주입 경로 | 모듈이 body만 출력하는 구조라면 title/description/OG/스키마를 넣을 별도 경로 확보 |
| 8 | 이미지 지연 로딩 + CDN | 에셋 많은 페이지 성능 관리 |
| 9 | 사이트맵 자동 갱신 | 신규 페이지 배포 시 자동 반영 |
| 10 | 리다이렉트/URL 설계 | 종료 상품 → 최신 페이지 자동 이동 |
| 11 | 검색엔진 등록 | 구글 서치콘솔 + 네이버 서치어드바이저 |

## 3. 텍스트 원고 작성 양식 (태그 지정 포함)

| 섹션 | 태그 | 텍스트 | 키워드 포함 |
|---|:---:|---|:---:|
| 히어로 | **H1** | (메인 키워드 포함 대제목) | ✅ 메인 키워드 |
| 히어로 | P | (부연 설명) | ✅ |
| 상세 섹션 1 | **H2** | (섹션 제목 — 연관 키워드) | ✅ 연관 키워드 |
| 상세 섹션 1 | P | (본문) | ✅ |
| 후기/사례 | **H2** | 후기 | ✅ |
| 후기/사례 | **H3** | (개별 후기 — 롱테일 키워드) | ✅ 롱테일 |
| CTA | **A(링크)** | (CTA 문구) → `/consult` | ✅ |

## 4. 성능·품질 목표

`../01-technical-seo/checklist.md` §5 기준 그대로: Lighthouse Performance 70점 이상, SEO 90점 이상, 에셋당 300KB 이하, 전체 페이지 10MB 이하.

**참고(경쟁사 벤치마크, KDT 사례)**: 텍스트가 전부 HTML로 출력되는 경쟁사는 90~100점을 기록한 반면, 플랫폼 한계(네이버 블로그 등)에 갇힌 채널은 50점대에 머물렀다. **모듈 구조 자체가 SEO 상한선을 결정**하므로, 태그 구조는 협의 우선순위 1번이다.

## 5. 모듈/개발 협의 시 필수 확인 질문

1. 텍스트 입력 시 `<h1>/<h2>/<p>` 같은 태그를 선택할 수 있는가?
2. 콘텐츠 이미지 에셋이 `<img>` 태그로 나오는가, CSS 배경으로 나오는가?
3. 이미지 에셋에 alt 입력칸이 있는가?
4. CTA 버튼이 `<a>` 태그 링크로 나오는가?
5. 섹션 블록 wrapper가 `<section>`인가, `<div>`인가?
6. 블록 조합 시 제목 순서(H1→H2→H3) 자동 검증 기능이 있는가?
7. head 정보(제목/설명/스키마)를 넣을 경로가 있는가?
