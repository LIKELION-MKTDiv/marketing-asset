페이지 최상단(`# CS 주간 지표` 헤딩 바로 위)에 아래 한 줄을 평문으로 고정 삽입한다. 콜아웃 아님, 매번 문구를 바꾸지 않음.

## 정성 분석 규칙 (필수 — 모든 섹션에 적용)

당신은 이 데이터를 6개월째 봐온 시니어 CX 분석가다. 집계와 분류는 분석이 아니다.
분석은 "왜 이 숫자가 나왔고, 그래서 무엇을 해야 하며, 안 하면 어떻게 되는가"까지 서술하는 것이다.

### 규칙 1. 이번 주 결론 한 문장 (리포트 최상단)
- 리포트 제목 바로 아래에 `> 🎯 **이번 주 결론:**` 콜아웃 1개를 둔다.
- 이번 주 데이터에서 가장 중요한 신호 딱 1개를 골라 판단을 내린다.
  모든 발견을 나열하지 말고 우선순위를 찍어라. 나열은 집계, 선택이 분석이다.
- 예: "지표는 전부 목표 내인데, 환불이 KDC·AIERS 두 입문 과정에 82% 쏠린 것이
  이번 주 유일하게 손대야 할 신호다."

### 규칙 2. 고객 원문(verbatim) 인용
- 매니저 연결 상위 3개 유형마다 고객이 실제로 쓴 문장을 2~3건 그대로 인용한다.
  매니저 시점 요약("환불 요청 → 처리 안내")만으로 끝내지 않는다.
- 형식: `💬 "…고객 발화 원문…" — {과정}, {MM/DD}`
- 이름·연락처·이메일·계좌 등 개인정보는 반드시 `[이름]`, `[연락처]` 형태로 마스킹한다.
- 원문이 확보되지 않은 유형은 인용을 지어내지 말고 "원문 미확보"로 표기한다.

### 규칙 3. 가설은 반드시 검증 방법과 세트로
- 원인 추정을 쓸 때는 아래 3줄 세트를 강제한다. 검증 방법 없는 추정은 쓰지 않는다.
  - **가설:** (예: KDC 환불은 가격 대비 기대치 불일치 때문이다)
  - **근거:** 이번 주 데이터에서 이 가설을 지지하는 구체적 숫자 또는 인용
  - **검증:** 다음 주에 확인 가능한 구체적 방법
    (예: 환불 9건의 채팅에서 환불 사유 발화 확인 + 결제 후 환불까지 경과일 분포)

### 규칙 4. 주차 간 이슈 추적 (연속성)
- `reports/` 디렉토리의 직전 1~2주 리포트를 반드시 읽고, 리포트에
  `## 이슈 추적` 섹션을 둔다. 각 항목은 다음 형식:
  - `{이슈명} — {N}주 연속 / 상태: 신규·지속·악화·개선·종결 / 지난주 액션: {실행 여부}`
- 지난주 액션 아이템이 이번 주 데이터에 반영됐는지 반드시 판정한다.
  (예: "AI 브랜드필름 일정 문의 — 3주 연속, FAQ 미반영 → 에스컬레이션 필요")
- 지난주에 세운 가설의 검증 결과를 이번 주 데이터로 판정해 기록한다.

### 규칙 5. 사업 이벤트와 결합
- `context/event_calendar.md`를 읽고, 숫자 변동을 이벤트와 연결해 해석한다.
  (개강일, 모집 마감, 광고 캠페인 시작, 가격/정책 변경 등)
- 예: "월요일 인입 43건"이 아니라 "GM06 모집 마감 직전 주말 누적분이
  월요일에 몰린 것"까지 써야 분석이다.
- 연결할 이벤트가 없으면 없다고 쓰고, 억지로 연결하지 않는다.

### 규칙 6. 금지 표현 (발견 시 해당 문장 재작성)
- 금지: "~인 것으로 보입니다", "~로 판단됩니다", "검토가 필요합니다",
  "지속 모니터링이 필요합니다", "개선 여지가 있습니다"
- 대체 원칙: 판단이면 근거 숫자·인용과 함께 단정하고,
  불확실하면 규칙 3의 가설-근거-검증 세트로 쓴다.
- "검토가 필요합니다" 대신 → 누가, 무엇을, 어떻게 확인할지 구체적 액션으로 쓴다.

**데이터 소스**: 기본은 `voc-report-automation/archive/voc_raw_{from}_{to}.csv` 재사용 (2026-07-09 확정, 토큰·API 절약 목적). 월요 자동화(`send_monday_report.py`)가 그 주 데이터를 이미 API로 수집해 이 파일에 저장·git 커밋해두므로, 화요판 작업 시 별도로 API를 다시 호출하지 않는다. 고객명·email만 제거돼 있고 태그·상담요약·state·응답시간·csat 등 분석에 필요한 컬럼은 전부 남아있음(고객명은 어차피 리포트에 못 쓰는 항목이라 손실 없음).
`collect.py` 재수집은 **그 파일이 없거나(자동화 실패·아직 미실행) 손상됐을 때만** 예외적으로 실행 — `voc-report-automation/collect.py --from YYYY-MM-DD --to YYYY-MM-DD`, 결과는 `voc-report-automation/voc_csv/`에 저장됨. Channel.io UserChat xlsx export(다운로드 폴더)는 보조/교차검증용으로만 사용 — `openpyxl`로 `data_only=True` 열어서 분석.

**API로 직접 재수집해야 할 때(`voc-report-automation/collect.py`, 예외 상황) 반드시 지킬 것**
- `state='closed'`로만 필터링하면 안 됨. ALF가 답변하고 고객이 더 응답하지 않으면 채널톡 상 채팅은 `closed`로 전환되지 않고 `initial`(대기) 상태로 그대로 남는다 — 즉 **ALF 단독해결 건의 상당수가 `initial` 상태에 머문다.**
  - 실사례(2026-07 첫째 주 검증): `state=initial & alfTriggered=true` 65건 vs `state=closed` 82건 — state 필터를 걸면 ALF 해결 건이 대량 누락되어 ALF 해결률이 실제보다 크게 낮게 계산된다(그 주 오분석: 7건/26.9%로 계산했다가, 원인 파악 후 정정).
  - ⚠️ `state` 파라미터를 아예 생략하는 것도 안 됨 — API가 파라미터 없이는 사실상 `opened` 소수 건만 반환한다(전체 상태 조회가 되지 않음). **`closed` / `initial` / `opened` 세 가지를 각각 명시해서 따로 페이징한 뒤 합쳐야** 한다.
  - `collect.py`의 `collect_chat_ids()`는 이 방식으로 이미 수정됨(2026-07-06, 3개 state를 순회하며 id를 set으로 합침) — 스크립트를 다시 손댈 때 `state='closed'` 단일 필터나 `state` 생략으로 되돌리지 말 것.
  - 수집 후 검증: 전화 태그(`전화`, `전화-착신X`) 제외한 비전화 채팅 총량이 대략 130~160건대(최근 주 기준)로 나오는지 확인 — 이보다 훨씬 적게 나오면(예: 80건대) state 수집이 또 깨진 것일 가능성이 큼.
  - xlsx export도 마찬가지: `state=initial`인 행을 무조건 "이탈건"으로 단정하지 말고 `alfTriggered` 값을 같이 확인할 것.
  - `collect.py` 출력 CSV의 `처리구분`에는 `전화`/`전화-착신X`도 섞여 있음 — ALF/매니저연결 분석 전에 반드시 이 두 값을 제외하고 볼 것 (전화 상담은 이 리포트 분석 대상이 아님).
  - **설문발송건 제외**: 상담요약이 "설문조사 안내 / 부트캠프 지원 경험에 대한 설문조사 진행 / 참여자 중 추첨을 통한 커피 쿠폰 제공" 같은 자동 설문 발송 문구면, 태그가 무엇이든(전화 태그가 아니어도) 실제 문의가 아니므로 분석 대상에서 제외할 것. 2026-07-06 검증 시점엔 이런 건이 전부 `전화` 태그로 잡혀 있어 이미 제외됐지만, 태그 없이 나올 수도 있으니 상담요약 텍스트로도 확인할 것.
  - **`상담요약` 추출 기준**: "마지막 bot 메시지"를 그대로 쓰면 안 됨 — 실제 처리 요약과 "추가로 궁금하신 점", "좋은 점수 감사합니다" 같은 CSAT/정형 안내 문구가 거의 같은 시각에 연속 발송되는 경우가 많아, 정형 문구가 뽑혀 실제 문의 내용이 사라질 수 있다 (2026-07-06 KDC 5건에서 확인됨). `collect.py`는 이런 정형 문구를 블록리스트로 걸러내고 그 이전의 실질 메시지를 요약으로 쓰도록 수정됨 — 되돌리지 말 것. 상담요약이 CSAT 안내 문구처럼 보이면 태그(2차 태그)나 원본 대화를 다시 확인할 것.
  - **`첫응답소요`(응답시간) 계산 기준**: "채팅의 맨 처음 고객 메시지"가 아니라 "매니저 첫 응답 직전의 마지막 고객 메시지"부터 재야 함. ALF가 여러 턴에 걸쳐 먼저 응대하다가 한참 뒤에야 고객이 매니저를 부르는 경우가 흔한데, 채팅 시작 시점을 기준으로 하면 ALF가 잘 응대하던 시간까지 "매니저 지연"으로 잘못 잡힌다 (2026-07-06 실사례: 10시간43분으로 집계된 건이 실제로는 매니저가 "매니저 연결해주세요" 요청 후 3분48초 만에 응답한 건이었음). `collect.py`는 이미 이 방식(`cust_ref_ms`, CSV의 `응답대상질문시각` 컬럼)으로 수정됨(2026-07-06) — 되돌리지 말 것.

**케이스 목록(지연 리스트·유형별 사례 등)에 개별 건을 나열할 때 지킬 것**
- **전화 관련 태그는 모두 제외**: `전화`, `전화-착신X`뿐 아니라 `전화안내`처럼 "전화"가 들어간 태그가 붙은 건은 채팅 케이스 목록에서 제외. (집계 대상 자체에서 빼는 `전화`/`전화-착신X`와 달리, `전화안내`는 실제 채팅이라 다른 집계에는 남지만 **케이스 나열에는 올리지 않음**.)
- **태그 미확인(빈 값/nan) 건은 리스트에서 삭제**: 과정을 특정할 수 없는 건은 사례로 보여주지 말 것.
- **`기타문의, KDC`처럼 기타문의+실제 과정 태그가 같이 붙은 건은 과정명만 표기**: "기타문의, KDC" → "KDC"로 표기 (위 과정별 인입 집계의 기타문의 재매칭 규칙과 동일한 원칙).
- **`멋사대학`은 케이스 목록에는 올리지 않되, 건수(집계)에는 포함**: 대학 문의는 별도 채널 안내 대상이라 사례로 상세히 다룰 필요는 없지만, 유형별 순위표 등 집계 수치에서는 계속 카운트할 것.

**3번(교육 매니저 연결 요청 심층분석) 유형 분류 표준 카테고리 — 매주 동일 기준으로 분류**
- 매니저 연결 건(`처리구분`='상담원연결')의 상담요약을 읽고 아래 10개 카테고리 중 하나로 분류. 새 유형이 필요하면 카테고리를 추가하되, 기존 이름은 바꾸지 말 것(주차 간 비교 가능하도록 고정):
  1. **환불/결제 처리 요청** — 환불 완료 안내, 영수증 발급, 결제수단 변경 재결제
  2. **클래스/과정 상세 안내** — 커리큘럼·일정·오프라인 여부 등 과정 자체에 대한 설명
  3. **결제/정책 확인** — 국취제, 내일배움카드 자부담금, 훈련장려금 등 복잡한 정책 확인
  4. **개인 상태·신청현황 확인** — 합격여부, 수강신청 처리여부, 선발상태 등 본인 계정 상태 확인
  5. **수강철회·취소·과정변경** — 수강 철회, 다른 과정으로 변경 희망
  6. **계정 처리** — 이메일 변경, 회원 탈퇴
  7. **시스템/기술 문제** — 카메라 미공유, 강의자료 접근 불가, 로그인 오류 등
  8. **오프라인 공간·지역 문의** — 오프라인 강의실 위치·이용 관련
  9. **단순 피드백(실질 문의 없음)** — CSAT 요청 자동 메시지 등 실제 문의 내용이 없는 건
  10. **기타** — 위에 안 맞는 소수 건
- 이 중 1·3·5·6은 "정책상 인적처리 필수"로 보고 ALF 해결률 계산의 구조적 제외 대상과 대략 대응함(위 **ALF 해결률 산정 방식** 섹션 참고) — 완전히 1:1은 아니므로 ALF 해결률용 구조적 제외는 별도로 케이스 재검토할 것.
- **"단순 피드백"으로 분류하기 전에 반드시 원본 대화(메시지 스레드)를 열어서 확인할 것**. `상담요약`이 "추가로 궁금하신 점..."이나 "좋은 점수 감사합니다..." 같은 CSAT/정형 문구로 보이더라도, 실제로는 그 직전에 진짜 처리 요약이 있고 태그(2차 태그)에도 실제 문의 성격이 남아있는 경우가 많다 — 태그만으로도 판단이 애매하면 API로 전체 메시지를 다시 불러와 확인. (`collect.py`의 상담요약 추출 로직은 이미 이 정형 문구를 걸러내도록 고쳐졌지만, 그래도 애매해 보이면 원본을 열어 재확인할 것.)
- **매니저 연결 요청 순위표(3번 섹션 표)는 상위 3~5개만 표시**. 전체 카테고리를 다 나열하지 말고, 비중이 큰 순서로 3~5개만 보여줄 것 — 나머지는 필요하면 본문 서술에서만 짧게 언급.

**TOP 3 상담 태그(과목) 상세 분류 서술 형식**

    - `**N. 과목명 (총 X건, 평균 Y)**`
    - **주요 문의 유형:** 그 과목 안에서의 세부 유형 상위 3개 + 건수 (예: 환불/취소 문의 9건, 등록·신청 상태 확인 4건, 클래스문의 1건) — 위 10개 표준 카테고리를 그대로 쓰거나, 과목 특성에 맞게 더 세분화해도 됨
    - **주요 문의 내용:** 실제 사례를 바탕으로 한 서술형 bullet 2~3개 (반복되는 패턴, FAQ 공백 신호, 특이사항 등)
- 세부 유형을 나눌 때 "CSAT 자동 피드백" 같은 걸 유형으로 만들지 말 것 — 위 "단순 피드백으로 분류하기 전에 원본 대화 확인" 규칙과 동일하게, 반드시 실제 대화 내용 기준으로 분류.

- "지연" 관련 신호는 **실제 응답시간(첫응답소요)이 확인된 장시간 지연 건에만** 붙일 것 — 키워드만 보지 말고 반드시 응답시간 수치와 함께 교차 확인.

**과정별 인입 문의 수 사전 집계 스크립트 (2026-07-10 추가)**
- `voc-report-automation/course_breakdown.py --from {from} --to {to}` — 아래 "과정별 인입 문의 수 집계 규칙"을 코드화해서, 태그로 과목이 이미 명확한 건만 자동 집계하고 표로 뽑아준다. 지난주(06/29~07/05) 데이터로 검증 완료 — 실제 화요판 표와 거의 일치(스냅샷 시점 차이로 인한 소폭 오차만 있었음, 로직 문제 아님).
- **경계(중요, 어기지 말 것)**: 이 스크립트는 태그가 이미 명확한 건의 "숫자 집계"만 자동화한 것이지, 판단·분류 작업을 대신하지 않는다. 아래는 여전히 사람이 원본을 읽고 직접 판단해야 하며 절대 생략하면 안 됨:
  - 스크립트가 뽑아주는 **review_queue(태그로 과목 특정 불가)** 목록 — 상담요약 원문을 읽고 실제 어떤 과정 문의인지, 혹은 진짜 무응답 이탈인지 판단
  - ALF 해결률 산정을 위한 매니저연결 건의 구조적 제외 여부 — 이건 `precise_review.py`(Claude 케이스별 검토, 캐시됨)가 담당하며 이 스크립트는 건드리지 않음. 화요판 작성 시 `precise_review_cache/{from}_{to}.json`을 참고할 것
  - CSAT/설문 정형문구 오분류, 중복 재오픈 여부, 응답지연 극단치의 실제 원인 — 전부 상담요약 원문 확인 필수
  - **"태그를 다시 매칭"하는 재분류 작업 자체는 스크립트가 아니라 사람이 하는 것** — 스크립트는 이미 붙은 태그를 그대로 신뢰해서 집계할 뿐, 태그 자체가 잘못 붙은 건은 스스로 걸러내지 못한다. 태그가 아예 없거나 과정 태그가 없는 건은 review_queue로 잘 걸러지지만(검증 시 확인함), **과정 태그가 하나라도 붙어있으면(그게 오분류든 아니든) 곧바로 그 과정으로 집계돼버림** — 즉 잘못된 태그가 붙은 건은 표 숫자에 섞여 들어가도 review_queue에 안 걸리고 조용히 지나갈 수 있다. 표 숫자를 과신하지 말고, 특정 과정 건수가 평소와 다르게 튀거나 하면 그 과정의 원본 상담요약을 샘플링해서 확인할 것.

**과정별 인입 문의 수 집계 규칙**
- 분모는 **실제 과목명(과정 태그)만** — 이메일_변경, 무태그, 결제환불, 회원정보/소셜로그인, 미래내일일경험, 멋사대학, KDT/지원완료 등 과목이 아닌 카테고리는 표에서 완전히 제외 (건수도 안 셈)

- `기타문의` 태그가 붙은 건은 같은 채팅에 붙은 **다른 태그(2번째 이후)가 있으면 그 태그로 재매칭**. 재매칭 후에도 남는 기타문의는 보통 `state=initial/waiting`(응대 없이 이탈한 건)일 확률이 높음 — 이 경우 과목 특정 불가로 집계 자체에서 제외

**ALF 해결률 산정 방식 (2026-07-10 전면 재정의)**

*0단계 — 모집단 확정*
- 전체 채팅 중 전화 계열(`처리구분`이 `전화`/`전화-착신X`)은 애초에 ALF를 거치지 않으므로 제외.
- **`처리구분`이 비어있고 `태그`도 없는 건은 전화 상담으로 간주해 제외**(2026-07-10 확정 — "알람 링크 전송" 같은 정형 요약은 전화 상담 시 보내는 문구로 확인됨).
- 남은 건(`ALF 해결` + `상담원연결`)이 **"ALF 상담 신청 건수"** — 여기서부터 분모·분자를 나눈다.

*1단계 — 분자: ALF 단독해결*
- `처리구분=ALF 해결` 건수를 기본 분자로 삼는다.
- 이 중 일부가 qna-test 슬랙 채널(아래 참고)에도 걸릴 수 있는데, **태그만 보고 자동으로 분자에서 빼지 말 것** — 상담요약을 읽어서 "고객이 문제 해결 완료" 같은 문구가 있으면 매니저 개입 없이 진짜 ALF가 해결한 것이므로 분자에 그대로 둔다.

- 이 채널은 `aitlab` 봇이 실제 매니저 연결 요청건마다 "채널톡 직접상담요청" 메시지로 **Channel.io 상담 URL(=chatId 포함)**을 남기는 채널. `SLACK_BOT_TOKEN`으로 `conversations.history` API 직접 호출해서 chatId를 추출·매칭 가능(채널ID: `C0B886S371Q`, MCP 도구 없이도 curl/requests로 접근 가능).
- **이전 지침("이 채널은 근거로 쓸 수 없음", 2026-07-07 검증 36% 정확도)은 폐기.** 2026-07-10 재검증(06/29~07/05 기준) 결과 매니저연결 79건 중 73건(92%)이 정확히 매칭됨 — chatId 기반 매칭은 신뢰할 만함. 다만 매주 정확도가 달라질 수 있으니 **매주 재검증할 것, 맹신 금지**.
- 주의점 2가지:
  - `ALF 해결` 태그건도 소수(이번 주 2/73) 이 채널에 걸릴 수 있음 — 대화 중 "매니저" 키워드만 언급되고 실제론 ALF가 해결한 경우. **반드시 상담요약을 읽어서 확인**하고, 자동으로 분자에서 빼지 말 것.
  - 매니저연결 79건 중 이 채널에 없는 소수(이번 주 6/79)도 있음 — 이건 2단계 카테고리 분류로 별도 판단.
- 사용법: 이 채널에 걸린 매니저연결 건은 구조적 제외로 간주하되, 2단계 카테고리 분류와 병행해서 교차검증할 것(둘 중 하나로만 결정하지 말 것).
- **익명 방문자(placeholder) 세션을 이유로 ALF해결 분자에서 제외하지 말 것**: `고객명`이 "단어+숫자3자리" 패턴(예: 슈팅스타 545, Maple 162)인 건 Channel.io가 미로그인 방문자에게 자동 부여한 닉네임일 뿐, 비회원·익명 방문자도 실제 문의를 할 수 있으므로 익명이라는 이유만으로 실질 문의가 아니라고 단정하면 안 됨(2026-07-07 오판 사례 — 한때 이 패턴 37건을 일괄 제외했다가 되돌림). 익명 세션도 다른 건과 동일하게 상담요약 내용을 보고 실질 문의 여부를 판단할 것.

*실제 계산 예시 (06/29~07/05)*
- ALF 상담 신청 152건(ALF해결 73 + 매니저연결 79) → 매니저연결 중 구조적 제외 55건, 분모에 남긴 것 24건(제도정책9+정보성9+기술지원3+콜백불명3)
- 분자 72건(ALF해결, 상담요약 확인 후 조정) / 분모 96건(72+24) = **75.0%**
- 목표는 70% 이상. **목표를 달성한 주에는 "미해결 건 증가" 식으로 알람성 헤드라인을 달지 말 것** — 진짜 미해결 건 분석은 참고 분석으로 톤을 낮춰서 심층분석 섹션에 배치
- `aitlab`(ITLAB AI 보조응대) 태그 부착 여부를 매주 체크: 신규 AI보조 응대가 실제로는 "매니저 직접 응대"로 기록되어 ALF 실패로 오집계될 수 있는 구조이므로, 태그가 실제로 붙고 있는지 xlsx에서 확인하고 미적용 시 재요청

**첫 응답 시간(운영시간 기준) 계산**
- 필드: `timeToFirstAnswerInOperation` (초 단위). 운영시간 10:00~18:00, 점심 12:00~13:00 제외, 주말 카운트 안 함
- 검증 예시: 금요일 17:55 인입 → 월요일 10:05 응답 = 금요일 잔여 5분 + 월요일 5분 = **10분** (주말/야간은 0으로 카운트, 계산 오류처럼 보여도 정상 로직)
- **극단치(며칠에 걸친 예외적 지연 등) 발견 시 평균 계산에서는 제외하되, 건수(n)에는 포함하고 TOP지연 리스트에는 남겨서 별도 원인 확인 액션아이템으로 처리**. 극단치 하나 때문에 특정 과정 전체가 문제 있는 것처럼 보이지 않게 주의
- **모든 응답시간 상위 케이스는 자동 제외 금지, 상담요약을 읽고 개별 판단**: 자동 산출값이 커도 실제로는 지연이 아닐 수 있음 — 상담요약에 "유선으로 안내 완료/전화로 처리" 같은 문구가 있으면 실제 응대는 채팅 밖(전화)에서 이미 끝났고 채팅상 응답시각은 사후 메모일 뿐이므로 제외 대상. "중복 채팅 건으로 앞선 상담창 안내" 같은 문구가 있으면 같은 문의의 재오픈 건이라 독립된 케이스로 세면 안 됨(2026-07-06 실사례: 임재현 고객 2건, 5시간대 응답으로 집계됐으나 각각 전화처리·중복채팅이었음). 단, "같은 고객이 N시간 내 재오픈"이라는 시간 간격만으로는 자동 판별 금지 — 같은 주에 정상적인 재문의도 다수 섞여 있어(2026-07-06 검증: 3시간 이내 재오픈 15쌍 중 진짜 중복은 일부) 반드시 상담요약 내용을 읽고 판단할 것.

**표현 관련**
- "무내용"이라는 단어는 쓰지 말 것 (사용자가 이 표현을 싫어함) — "구체적 질문 없이 제출된 건", "질문 내용 없이 접수된 건" 등으로 풀어서 쓸 것.
- **리포트(Notion 포함)에 고객 이름을 절대 기재하지 말 것**: "대표 사례" 등에서 특정 케이스를 예시로 들 때도 "임재현(발급지연)"처럼 이름+사유를 함께 쓰지 말고 "발급지연"처럼 사유·유형만 쓸 것(2026-07-07 위험사례 — 노션 페이지에 실명이 들어갔다가 발견되어 수정함). 채팅 중 분석 과정에서는 이름을 참고해도 되지만, 최종 발행물(마크다운/Notion/Slack)에는 절대 남기지 말 것.

**"주간 개선 추적" 섹션 형식 (문서 맨 끝, 매주 누적)**
- 구조: **지난주 문제**(1~2줄) → **이번주 결과**(3~4줄, 이번 주 핵심 지표 변화) → **개선 체크포인트 [진행완료]**(주차별 `<details><summary>` 토글로 누적) → **다음 주 체크포인트**
- `<details><summary>MM월 N주차 (MM.DD~MM.DD)</summary>` 토글로 지난 주차들을 접어서 쌓아올릴 것 — 주차 번호 옆에 항상 그 주의 날짜 범위를 표기(예: "6월 3주차 (06.22~06.28)"). 발행일이 여러 날 걸쳐 있거나 발행이 누락된 주가 있어도 "3~4주차"처럼 묶어 부르지 말고, 주차마다 각각 번호를 매길 것(한 주 = 한 번호).
- 지난 주차는 그대로 두고, 이번 주 발행 시 새 토글 하나만 맨 아래(다음 주 체크포인트 바로 위)에 추가. 과거 토글 내용을 요약하거나 삭제하지 말 것(누적 히스토리이므로).
- **주차 번호 기준일: 6월 1주차 = 06.01~06.07** (2026-06-09 발행분, 구버전 포맷이라 이 누적 로그에 처음엔 누락됐다가 2026-07-07에 뒤늦게 추가됨). 이후 매주 +1주씩 순차 증가 — 2026-07-07 기준 6월 5주차(06.29~07.05)까지 있음.
- 다음 신규 토글(06.29~07.05 다음 주, 즉 07.06~07.12)을 "7월 1주차"로 새로 시작할지 "6월 6주차"로 이어갈지는 아직 정해지지 않음 — 작성 시점에 화면에 보이는 최근 토글 번호를 보고 자연스럽게 이어가되, 애매하면 사용자에게 확인할 것.
- 새로 추가하는 토글의 체크박스는 그 주에 실제로 완료된 항목만 `[x]`로 표시하고, 미완료는 `[ ]`로 남겨서 다음 주로 넘길 것.
- 이 형식은 실제 2026-06-23 화요판부터 적용되기 시작함 — 그 이전 주차(5월 등)는 토글 없이 단순 들여쓰기 목록이었으나, 앞으로는 토글 형식으로 통일.

**전체 배치 순서 (중요)**: 헤드라인 트렌드바(첫응답시간/ALF해결률) → 누적요약 → 🟢1.첫응답시간/🟢2.ALF해결률 짧은요약 → **차트 3개 + 과정별 인입문의수 표** → 심층분석(3번~) 순서. 차트 묶음은 헤드라인 바로 아래가 아니라 짧은요약(🟢1/🟢2) 다음에 옴.

**콤보 차트 3개 (🟢2.ALF해결률 요약 다음, 심층분석 앞)**
- 막대(ALF 총 건수) + 선(첫응답시간 분:초)을 01/12부터 이번 주까지 누적으로 보여주는 이미지
- `VOC/tuesday/generate_trend_chart.py`의 `WEEKS`/`COUNTS`/`RESP_SEC`에 이번 주 값 한 줄 추가하고 `OUT_DATE` 갱신 후 실행 → `YYYY-MM-DD_trend_chart.png` 생성
- 라벨은 `분:초`(예: 5:47) 형식 유지 — 값이 가까운 주(예: 5분대·6분대)도 구분되도록. x축 라벨은 45도 대각선 회전 유지

**위 차트 바로 아래에 두 번째 콤보 차트 첨부 (막대: ALF 총 인입 건수 / 선: ALF 해결률 %)**
- `VOC/tuesday/generate_resolution_chart.py`의 `WEEKS`/`COUNTS`/`RATE`에 이번 주 값 한 줄 추가하고 `OUT_DATE` 갱신 후 실행 → `YYYY-MM-DD_resolution_chart.png` 생성
- 막대 y축은 `BAR_AXIS_MAX=180`으로 고정 (인입건수가 이 값을 넘어서면 다시 조정), 선 y축은 0~100% 고정 + 목표선 70% 점선 표시
- % 라벨은 각 점 위에 흰 배경 박스로 표시 (막대와 겹쳐도 읽히도록)
- COUNTS와 RATE는 각 주 리포트에 실제 발행됐던 값을 그대로 사용 — 과거 주차는 %와 "X/Y건" 분수가 정확히 일치하지 않을 수 있음(해결률 산정 방식이 최근에 엄격해졌기 때문). 억지로 맞추지 말 것

**위 두 차트 아래에 세 번째 차트 첨부 (이번 주 일자별, 월~일)**
- `VOC/tuesday/generate_daily_chart.py`의 `DAYS`/`COUNTS`/`RESP_SEC`를 이번 주 값으로 교체하고 `OUT_DATE`/`WEEK_LABEL` 갱신 후 실행 → `YYYY-MM-DD_daily_chart.png` 생성
- COUNTS = 그날 `timeToFirstAnswerInOperation`이 non-null인 채팅 수, RESP_SEC = 그날 값들의 평균(초, 극단치 제외)
- 데이터 없는 요일은 `None`으로 표기 (0으로 넣으면 "지연 없음"으로 오독됨) — 차트에 "데이터 없음"으로 표시됨
- 라벨 배치: 건수(파란색)는 막대 위, 응답시간(파란색 아님·빨간색)은 선 아래 — 두 라벨이 서로 다른 밴드에 있어 겹치지 않음. 0초처럼 축 바닥에 가까운 값은 예외적으로 위쪽에 표기

**"지난주" 비교 시 필수 확인**
- 트렌드 바 차트의 바로 위 행(가장 최근 완료 주)을 직접 보고 대입 — 기억이나 이전 리포트에 적혀있던 값을 그대로 재사용하지 말 것 (오타/구버전 값 혼용 사고 방지)
